PayLink — Payment & Invoice Management System

GitHub
Problem
Small businesses in Kenya lose financial control due to manual tracking of payments via WhatsApp, paper records, and spreadsheets with no enforcement of transaction integrity.
System Design
Backend-driven invoicing system enforcing a strict financial transaction lifecycle: Draft → Sent → Paid → Overdue → Reconciled. State transitions are validated at the model level with database constraints ensuring no invalid state progression.
Technical Depth
  • Financial state machine with strict invoice state transitions enforced at the model level, preventing illegal transitions (e.g., Paid → Draft)
  • Idempotency logic via unique idempotency keys on payment endpoints to prevent duplicate or double payment processing in concurrent scenarios
  • Audit-safe transaction records with immutable ledger entries; zero silent data mutation after record creation
  • Payment callback handling with signature verification and replay attack prevention via nonce tracking
  • PDF invoice generation using headless rendering with transactional consistency (PDF generation only occurs on Paid state, stored as immutable artifact)
  • Email notification system via Mailtrap for payment confirmations and overdue alerts
Production Readiness
MySQL transactions Race condition prevention Database-level constraints Idempotent API design

EduCore — Multi-Tenant School Management SaaS

GitHub
Problem
Schools operate using fragmented manual systems with no centralized academic data structure, leading to data silos and inconsistent student records across terms.
System Design
Multi-tenant SaaS where multiple schools operate on a single shared codebase and database. Tenant isolation enforced at query level via global scopes, not just application logic, with every request scoped to a tenant identifier derived from subdomain or JWT claim.
Technical Depth
  • Tenant isolation at query level using Laravel global scopes ensuring no cross-tenant data leakage even in edge cases (e.g., direct ID access attempts)
  • Role-based access control (RBAC) combined with tenant scoping; permissions evaluated per-tenant preventing privilege escalation across tenants
  • Background job tenant scoping ensuring queue workers process only tenant-specific jobs with correct database context
  • Shared schema multi-tenancy with tenant_id on all models and composite indexes for tenant+entity queries
  • Academic year partitioning within tenant scope for efficient historical data queries
Production Readiness
Tenant data isolation RBAC with tenant context Queue worker scoping Database indexing strategy

OrphanageTracker (OT System) — Student Accountability System

GitHub Deployment
Problem
Orphanage institutions lack structured systems for tracking students, staff activity, and documentation, relying on paper files and ad-hoc spreadsheets.
System Design
Role-based management system with hierarchical permissions (super_admin, staff, student) and document storage with structured metadata for audit trails.
Technical Depth
